# AECC · Нагрузочный клиент-серверный сессионный тест

Дополнение к базовому тест-киту (`AECC_TestServer` / `AECC_TestClient`). Три новых куска:

| Папка | Что это |
|---|---|
| `tests/AECC_LoadShared/` | Общие исходники (константы `LK`, компоненты, события) — линкуются **в оба** проекта |
| `tests/AECC_LoadServer/` | Авторитарный сервер: сессии, GDAP, мины (DBComponent), роллинг, двойные проверки |
| `tests/AECC_LoadClient/` | **Мультиклиент-хост**: N виртуальных клиентов, каждый со своим TCP-соединением |

## Запуск

```bash
dotnet run -c Release --project tests/AECC_LoadServer   [timeoutSec=300]
# затем, вторым процессом:
dotnet run -c Release --project tests/AECC_LoadClient   [clients=8] [durationSec=75] [prefix]
```

Exit code 0 у обоих — все проверки прошли (мультиклиент дополнительно шлёт свой отчёт
серверу событием `LoadReportEvent`, сервер сводит общий итог).

Можно запускать **несколько** мультиклиент-процессов против одного сервера (задайте
разные `prefix`) — тогда по-настоящему заработает удаление чужих сущностей из
клиентского мира по `SessionMemberLeftEvent` (в одном процессе все игроки «свои»,
и хост держит их сущности живыми по интерес-рефкаунту).

## Константы (`tests/AECC_LoadShared/LoadKit.cs`)

Всё, что просила постановка, — статические поля `LK.*`, переопределяются окружением
`AECC_LOAD_<ИМЯ>` без пересборки (сервер и клиент должны видеть одинаковые значения):

| Константа | Смысл | Дефолт |
|---|---|---|
| `MaxUsersOnServer` | максимум пользователей на сервере (сверх — `SERVER_FULL`) | 24 |
| `MaxUsersPerSession` | максимум пользователей в сессии | 4 |
| `MaxOperationalComponents` | ёмкость операционных компонентов (задел) | **32** (const) |
| `GunsPerPlayer` | активных пушек у игрока (≤ 32) | 4 |
| `ShotsPerSecond` | частота сессионной операции на игрока, распределяется по всем пушкам | 6.0 |
| `MaxSessionsOnServer` | сессий генерируется на старте сервера | 6 |
| `SessionKillTargetMin/Span` | случайный килл-каунт сессии, **не менее 100** | 100 + rand(40) |
| `SessionRestartMs` | перезапуск сессии | 5000 |
| `UpgradeCostGrowth` | прогрессия цены прокачки (бесконечно) | **×1.1** |
| `PlayerMaxHp` | ХП (одно на всех) | 100 |
| `MinePlacesPerSecond` | откат способности «мина» | 7/с |
| `MineFuseMinMs/MaxMs` | случайный взвод мины | 1000..2000 |
| `MineDamage` | урон мины всем игрокам сессии | 20 |
| `MulticlientCapacity` | сколько клиентов обеспечивает хост единовременно | 24 |
| `VerifyMode` | **двойные проверки клиент⇄сервер** | true |

`AECC_LOAD_VERIFYMODE=false` — «истинная нагрузочная способность»: отключает
wire-инспекцию GDAP, `StateCheck`-циклы, `ShootRejected`-ответы и сверки прогнозов.

## Модель

- **Сервер — авторитет.** Клиент только шлёт бизнес-события (`ShootEvent`,
  `PlaceMineEvent`, `BuyUpgradeEvent`, `Join/LeaveSessionEvent`); в данные (урон,
  откат, киллы, золото) их превращает сервер с двойной проверкой (откат пушки,
  принадлежность к сессии, ёмкость сессии, достаточность золота).
- **Роллинг.** `LoadRollEvent : UpdateEntitiesEvent` — GDAP-срезы по
  интерес-множеству получателя: своя сущность + **все** сессии (постоянно
  обновляемая «карта сессий») + участники своей сессии. Полный снапшот — на
  HELLO и на вход в сессию.
- **События вместо роллинга — только там, где роллинг бессилен**: выход
  участника из видимости (`SessionMemberLeftEvent`), рестарт с раздачей золота
  (`SessionRestartingEvent`), вердикты живости.

### GDAP (одна `LoadReplicationPolicy` на сущность)

| Компонент | Available (только владелец) | Restricted (только остальные) |
|---|---|---|
| `PlayerPublicComponent` | ✔ | ✔ |
| `GunReloadComponent`, `MineAbilityComponent` (перезарядки) | ✔ | — |
| `GunsComponent`, `GoldComponent` | ✔ | — |
| `HpComponent` | — | ✔ |
| `SessionInfo/Modifier/MinesDB` (на сессии) | — | ✔ |

Это в точности требование: перезарядку видит только сам игрок; ХП видят все,
**кроме** самого игрока (асимметрия Available/Restricted).

### Мины (тест `DBComponent`)

`MinesDBComponent : ComponentsDBComponent` на сущности сессии; строки
(`MineComponent`) принадлежат сущностям игроков через `IECSObjectPathContainer`.
Взвод 1–2 c → сервер взрывает (урон всем участникам с модификатором сессии) и
удаляет строку (клиентам доезжает `state=Removed`). Смерть игрока → его
неразорвавшиеся мины `RemoveComponentsByOwner` — исчезают не взрываясь.

### Реестр живости (`IECSObjectPath`)

Хост ведёт учёт мин (по строкам DB) и чужих сущностей: подозрительные (живут
дольше `LivenessSuspectAgeMs`) плюс небольшая плановая выборка каждые
`LivenessSweepMs` сверяются с сервером (`LivenessQueryEvent` →
`LivenessVerdictEvent`); мёртвое вычищается локально. Это ловит именно висящие
инстансы: строки мин ушедших игроков и сущности, оставшиеся после выхода из сессии.

### Мультиклиент и общий мир — важная оговорка

Конверт роллинга и path-контейнеры DB несут **сериализуемый** `ECSWorldOwnerId`,
который обязан резолвиться в локальный мир по общей константе (`LK.WorldId`), а
`WorldRegistry` процесса ключуется по id ⇒ N миров с одним id в одном процессе
невозможны. Поэтому виртуальные клиенты хоста разделяют **одну** клиентскую
реплику (заодно N приёмных потоков конкурентно бьют в одни `StabilizationGate` —
полезный стресс). Следствие: приватность GDAP проверяется **на проводе** —
verify-режим инспектирует blob'ы каждого `LoadRollEvent` per-получатель
(перезарядка — только в blob'ах владельца; свой Hp — никогда владельцу), а не по
содержимому общего мира.

## Что проверяется (verify mode)

Сервер: инварианты (hp в диапазоне, экономика «выдано == на руках + потрачено»),
прогнозы клиента (цена/золото), `StateCheck`-сверки, отсутствие жёстких отказов.
Клиент: wire-GDAP без нарушений, приватная доставка перезарядок, живая «карта
сессий», появление/исчезновение мин в DB, сходимость экономики, работа реестра
живости, отсутствие древних неразрешённых подозреваемых.
